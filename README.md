# Provenance Guard

Provenance Guard is a Flask backend API that analyzes submitted text to
determine whether it is more likely **written by a human** or **generated by
AI**. It combines two independent detection signals, reports a calibrated
confidence score, returns a plain-language transparency label, records every
decision in an audit log, and lets creators appeal a classification.

The goal is not perfect AI detection (an unsolved research problem) but a
**transparent, honest** attribution system that communicates uncertainty and
gives creators recourse.

## Features

- **Content submission** — `POST /submit` accepts text + `creator_id`, validates
  input, and returns a classification with a UUID `content_id`.
- **Multi-signal detection** — a Groq LLM signal and a deterministic stylometry
  signal run independently and are combined.
- **Confidence scoring** — the two signals are blended into a single, calibrated
  confidence value.
- **Transparency labels** — every result carries one of three plain-language
  labels (high-confidence AI, high-confidence human, or uncertain).
- **Appeals workflow** — `POST /appeal` lets creators contest a decision; the
  original audit record is moved to `under_review`.
- **Rate limiting** — `POST /submit` is throttled per IP to deter abuse.
- **Audit logging** — every decision and appeal is persisted with a full,
  structured schema.
- **Robust error handling** — 400 / 404 / 429 / 500 responses with clear JSON.

## Detection signals

**Signal 1 — Groq LLM (`llama-3.3-70b-versatile`), weight 70%.** Uses semantic
reasoning (coherence, style, tone, fluency) to predict `Human` or `AI` with a
confidence, converted internally to a probability of being AI-generated
(`p_ai`).

**Signal 2 — Stylometric heuristics, weight 30%.** Deterministic Python that
averages four features into `p_ai`:

- **Sentence-length uniformity** — low variation leans AI.
- **Contraction usage** — absence of contractions leans AI.
- **Lexical diversity** — low type-token ratio leans AI.
- **Average word length** — elevated length leans AI.

Each signal is independent, so a single failure mode is unlikely to dominate.

## Confidence scoring

Both signals produce `p_ai` (probability the text is AI-generated). They are
blended with a weighted average:

```
combined_p_ai = (groq_p_ai × 0.70) + (stylometric_p_ai × 0.30)
```

- Final prediction: **Likely AI** if `combined_p_ai >= 0.50`, else **Likely
  Human**.
- Confidence: the strength of the winning side,
  `max(combined_p_ai, 1 - combined_p_ai)` — always in **0.50–1.00**.
- `groq_score` and `stylometric_score` in the audit log are each signal's score
  re-expressed toward the winning prediction.

A single threshold of **0.90** separates a high-confidence label from an
uncertain one, so borderline cases are reported honestly rather than forced into
a definitive verdict.

## Transparency labels

Exactly one of three labels is returned. The text below is exactly what the API
returns (see `labels.py`).

### `high_confidence_ai`

Condition: prediction is `Likely AI` **and** confidence `>= 0.90`.

> Likely AI-generated. Our analysis strongly suggests this content was generated using AI. Creators may submit an appeal.

### `high_confidence_human`

Condition: prediction is `Likely Human` **and** confidence `>= 0.90`.

> Likely Human-written. Our analysis strongly suggests this content was written by a human author.

### `uncertain`

Condition: everything else.

> Uncertain Attribution. Our system could not confidently determine whether this content was AI-generated or human-written.

## Rate limiting

`POST /submit` is limited to **`10 per minute; 100 per day` per IP address**
using [Flask-Limiter](https://flask-limiter.readthedocs.io/) keyed on the remote
address, with in-memory storage (`storage_uri="memory://"`). This throttles
automated abuse while allowing normal use. Exceeding the limit returns HTTP
`429`. `GET /log`, `POST /appeal`, and `GET /` are **not** rate limited.

> In-memory storage resets on restart and is per-process; a production
> deployment with multiple workers should use a shared backend such as Redis.

## Appeals workflow

A creator who believes their content was misclassified submits `content_id` and
`creator_reasoning` to `POST /appeal`. The system validates both fields, finds
the matching audit record (`404` if absent), and updates **the same record** in
place: `status` → `under_review`, `appealed` → `true`, `appeal_reasoning` → the
explanation. No automatic reclassification occurs — the original decision is
preserved for human review and remains visible in `GET /log`.

## Audit log

Every submission is appended to `audit_log.json` with these fields:

| Field                | Meaning                                          |
| -------------------- | ------------------------------------------------ |
| `content_id`         | UUID for the submission                          |
| `creator_id`         | Caller-supplied creator identifier               |
| `timestamp`          | UTC ISO-8601 (`...Z`) time of classification     |
| `prediction`         | `Likely Human` / `Likely AI`                     |
| `confidence`         | Overall confidence (0.50–1.00)                   |
| `groq_score`         | Groq signal aligned to the final prediction      |
| `stylometric_score`  | Stylometric signal aligned to the prediction     |
| `transparency_label` | Label type applied                               |
| `status`             | `classified` or `under_review`                   |
| `appealed`           | `false` until an appeal is filed                 |
| `appeal_reasoning`   | `null` until an appeal is filed                  |

The raw submitted text is deliberately not stored. Appeals update the existing
record rather than creating a new one.

## API documentation

### `GET /`

Health check → `{ "status": "running" }`.

### `POST /submit`

Classify a block of text. **Rate limited to `10 per minute; 100 per day`.**

**Request**

```json
{ "text": "Some text...", "creator_id": "user123" }
```

**Response**

```json
{
  "content_id": "6f1c2e5a-8d3b-4a2f-9c11-2b7d0e4a1f33",
  "prediction": "Likely Human",
  "confidence": 0.84,
  "label": {
    "type": "high_confidence_human",
    "text": "Likely Human-written. Our analysis strongly suggests this content was written by a human author."
  }
}
```

### `POST /appeal`

Open an appeal against a prior classification. Both fields required.

**Request**

```json
{ "content_id": "uuid", "creator_reasoning": "I wrote this myself." }
```

**Response**

```json
{ "message": "Appeal received.", "status": "under_review" }
```

Returns `404` if the `content_id` is not found.

### `GET /log`

Return every audit entry, including appeal updates. **Not rate limited.**

```json
{
  "entries": [
    {
      "content_id": "...",
      "creator_id": "user1",
      "timestamp": "2026-07-01T22:10:00Z",
      "prediction": "Likely Human",
      "confidence": 0.92,
      "groq_score": 0.95,
      "stylometric_score": 0.88,
      "transparency_label": "high_confidence_human",
      "status": "classified",
      "appealed": false,
      "appeal_reasoning": null
    }
  ]
}
```

## Error handling

| Situation              | HTTP  | Body                                          |
| ---------------------- | ----- | --------------------------------------------- |
| Invalid JSON body      | `400` | `{"error": "Invalid JSON body"}`              |
| Missing `text`         | `400` | `{"error": "Field 'text' is required"}`       |
| Missing `creator_id`   | `400` | `{"error": "Field 'creator_id' is required"}` |
| Missing appeal fields  | `400` | `{"error": "Field '...' is required"}`        |
| `content_id` not found | `404` | `{"error": "content_id not found"}`           |
| Groq API failure       | `500` | `{"error": "Groq service unavailable"}`       |
| Rate limit exceeded    | `429` | Flask-Limiter default response                |

## Known limitations

- **Detection is heuristic, not definitive.** Neither the LLM nor stylometry can
  reliably identify AI text; results are probabilistic and can be wrong.
- **JSON-file storage.** The audit log is a single JSON file — not concurrency-
  safe under parallel writes and not suitable for high throughput. SQLite/
  Postgres is planned.
- **In-memory rate limiting.** Limits reset on restart and are per-process; a
  multi-worker deployment needs a shared backend (Redis).
- **No authentication.** `creator_id` is trusted as provided; anyone can submit
  or appeal.
- **English-oriented stylometry.** Contraction and word-length heuristics assume
  English; other languages will be scored less meaningfully.
- **Groq dependency.** If the Groq API is unavailable, `/submit` returns `500`;
  there is no offline fallback signal.
- **Short-text neutrality.** Very short inputs yield a neutral stylometric score,
  so confidence leans on the LLM alone.

## Future improvements

- **Ensemble detection** — add a third independent signal (e.g. perplexity or a
  second model) and vote across signals.
- **Durable storage** — replace the JSON file with SQLite/Postgres for
  concurrency safety and querying.
- **Shared rate-limit backend** — move to Redis so limits hold across workers.
- **Authentication & authorization** — verify `creator_id` and restrict appeals
  to the original submitter.
- **Confidence calibration** — tune weights and thresholds against a labeled
  dataset instead of fixed values.
- **Reviewer tooling** — an admin view to resolve `under_review` appeals.
- **Multi-language stylometry** — language-aware heuristics beyond English.

## Spec reflection

- **What matched the spec:** the endpoints (`/submit`, `/appeal`, `/log`), the
  multi-signal pipeline, the three transparency labels (exact text), the
  appeal-in-place update, rate limiting on `/submit` only, and the complete
  audit-log schema were all implemented as specified.
- **Where the design evolved:** the original plan weighted Groq/stylometry
  60/40; the implementation uses **70/30** because the LLM proved the stronger
  signal in testing, so `planning.md` was updated to match. The plan also
  sketched multiple confidence bands, but only the **0.90** threshold affects the
  transparency label, so the confidence section was consolidated to reflect the
  actual behavior.
- **What I would do next:** move storage to SQLite, add a shared rate-limit
  backend, add authentication, and calibrate confidence against a labeled
  dataset.
