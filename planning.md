# Provenance Guard — Planning

## Project Overview

Provenance Guard is a Flask backend API that helps determine whether submitted
text is more likely written by a human or generated by AI. The system combines
two independent detection signals, calculates a confidence score, generates a
transparency label for users, stores every attribution decision in an audit
log, and provides an appeals workflow for creators who believe their work was
incorrectly classified.

The goal is not perfect AI detection — an unsolved research problem — but a
trustworthy, transparent attribution system that communicates uncertainty
honestly while giving creators a mechanism to challenge decisions.

> **Note:** This document is kept consistent with the implemented code. Where a
> design idea evolved during implementation (for example the exact signal
> weights), this plan reflects what the code actually does.

---

# Architecture Narrative

## Submission Flow

When a user submits text to `POST /submit`, the system first validates the
request (both `text` and `creator_id` are required, and the body must be valid
JSON). The validated text is then passed to two independent detection signals:

1. **Groq LLM Classification** — uses the Groq API with the
   `llama-3.3-70b-versatile` model to decide whether the writing appears more
   likely human-written or AI-generated, returning a prediction and confidence.
2. **Stylometric Heuristic Analysis** — computes deterministic writing
   statistics and produces a probability that the text is AI-generated.

Both outputs are expressed as `p_ai` (probability the text is AI-generated) and
sent to the **Confidence Calculator**, which blends them with a weighted average
into a single prediction and confidence score. The **Transparency Label
Generator** converts that result into a user-facing message. Finally, the
complete decision is written to the **audit log** before the JSON response is
returned.

## Appeal Flow

If a creator believes the system misclassified their work, they submit an appeal
to `POST /appeal` with the `content_id` and their `creator_reasoning`. The
system locates the matching audit record, updates its status to `under_review`,
sets `appealed` to `true`, stores the appeal reasoning, and returns
confirmation. No automatic reclassification occurs; the original decision is
preserved for human review.

# Detection Signals

## Signal 1 — Groq LLM Classification

- **Purpose:** Use the Groq `llama-3.3-70b-versatile` model to judge whether
  submitted text appears human-written or AI-generated using semantic reasoning.
- **Measures:** semantic coherence, logical flow, writing style, tone, and
  language consistency — high-level characteristics that are hard to capture
  with simple statistics.
- **Output:** `prediction` (`"Human"` or `"AI"`) and `confidence` (0.00–1.00),
  converted internally to a `p_ai` probability.
- **Weight:** **70%**
- **Blind spot:** The LLM may classify exceptionally polished human writing as
  AI-generated, or sophisticated AI writing as human. It should therefore never
  determine the final decision alone.

## Signal 2 — Stylometric Heuristics

- **Purpose:** Measure structural writing characteristics with deterministic
  Python code, providing an independent signal that complements the LLM.
- **Metrics (as implemented in `stylometry.py`):**
  - **Sentence-length uniformity** — coefficient of variation of sentence
    lengths; low variation leans AI.
  - **Contraction usage** — frequency of contractions (e.g. `don't`, `we'll`);
    absence leans AI.
  - **Lexical diversity** — type-token ratio; low diversity leans AI.
  - **Average word length** — elevated average leans AI (formality).
- **Output:** `stylometric_p_ai` (0.00–1.00) — the average of the four feature
  scores.
- **Weight:** **30%**
- **Blind spot:** Professional authors often write consistently, while modern AI
  can intentionally introduce variability, so stylometry alone is unreliable.

## Combining Signals

Each signal is expressed as `p_ai`. The combined probability is a weighted
average:

```
combined_p_ai = (groq_p_ai × 0.70) + (stylometric_p_ai × 0.30)
```

The final prediction is derived from the combined score, not either signal
alone:

- `combined_p_ai >= 0.50` → **Likely AI**
- otherwise → **Likely Human**

Combining two independent signals reduces reliance on any single method and
produces a more reliable confidence score.

---

# Confidence Scoring Strategy

Confidence is the strength of the winning prediction, i.e.
`max(combined_p_ai, 1 - combined_p_ai)`, so it always falls in **0.50–1.00**.

A single threshold drives the transparency label:

| Combined confidence | Interpretation  | Transparency label            |
| ------------------- | --------------- | ----------------------------- |
| `>= 0.90`           | High confidence | `high_confidence_ai` / `human` |
| `< 0.90`            | Uncertain       | `uncertain`                   |

Reporting confidence — rather than pretending every prediction is definitive —
reduces false positives and encourages cautious interpretation. A result near
0.50–0.60 typically means the two signals disagree or neither provides strong
evidence, so the system shows an **Uncertain** label instead of a definitive
attribution.

## False Positive Analysis

False positives (labeling human work as AI) are the most harmful failure mode.
Example: a human author submits an original short story; Groq predicts AI with
moderate confidence while stylometry is mixed. Because the combined confidence
stays below 0.90, the system returns an **Uncertain** label rather than a
definitive AI verdict, and the creator may appeal. This protects creators from
unfair automated decisions.

---

# Transparency Labels

Exactly one of three labels is returned. The text below matches `labels.py`.

## High-Confidence AI (`high_confidence_ai`)

Condition: `prediction == "Likely AI"` and `confidence >= 0.90`.

> Likely AI-generated. Our analysis strongly suggests this content was generated
> using AI. Creators may submit an appeal.

## High-Confidence Human (`high_confidence_human`)

Condition: `prediction == "Likely Human"` and `confidence >= 0.90`.

> Likely Human-written. Our analysis strongly suggests this content was written
> by a human author.

## Uncertain (`uncertain`)

Condition: everything else.

> Uncertain Attribution. Our system could not confidently determine whether this
> content was AI-generated or human-written.

# API Design

## GET /

Health check.

**Response**

```json
{ "status": "running" }
```

## POST /submit

Analyze submitted text. Rate limited to `10 per minute; 100 per day` per IP.

**Request**

```json
{
  "text": "Your content goes here.",
  "creator_id": "user123"
}
```

**Response**

```json
{
  "content_id": "6f1c2e5a-8d3b-4a2f-9c11-2b7d0e4a1f33",
  "prediction": "Likely Human",
  "confidence": 0.84,
  "label": {
    "type": "high_confidence_human",
    "text": "Likely Human-written. Our analysis strongly suggests this content was written by a human author."
  }
}
```

## POST /appeal

Submit an appeal against a prior classification.

**Request**

```json
{
  "content_id": "6f1c2e5a-8d3b-4a2f-9c11-2b7d0e4a1f33",
  "creator_reasoning": "This article was written entirely by me."
}
```

**Response**

```json
{ "message": "Appeal received.", "status": "under_review" }
```

Returns `404` if `content_id` is not found.

## GET /log

Retrieve the full audit history (not rate limited).

**Response**

```json
{
  "entries": [
    {
      "content_id": "...",
      "creator_id": "user1",
      "timestamp": "2026-07-01T22:10:00Z",
      "prediction": "Likely Human",
      "confidence": 0.92,
      "groq_score": 0.95,
      "stylometric_score": 0.88,
      "transparency_label": "high_confidence_human",
      "status": "classified",
      "appealed": false,
      "appeal_reasoning": null
    }
  ]
}
```

# Appeals Workflow

Any creator whose content has been analyzed may appeal. The appeal must contain
`content_id` and `creator_reasoning`. When an appeal is submitted the system:

1. Validates the request (both fields required).
2. Locates the matching audit record (404 if not found).
3. Updates the same record in place: `status` → `under_review`, `appealed` →
   `true`, `appeal_reasoning` → the creator's explanation.
4. Returns confirmation.

A human reviewer can then see, from the audit record: the original prediction,
Groq score, stylometric score, final confidence, transparency label, appeal
reasoning, and current status.

---

# Rate Limiting Strategy

`POST /submit` is limited to **10 requests per minute and 100 per day per IP
address** using Flask-Limiter with in-memory storage. This supports normal
platform usage while reducing the risk of automated abuse. `GET /log`,
`POST /appeal`, and the health check are not rate limited.

---

# Audit Log Design

The audit log (`audit_log.json`) stores one record per submission with the
following fields:

- `content_id`
- `creator_id`
- `timestamp`
- `prediction`
- `confidence`
- `groq_score`
- `stylometric_score`
- `transparency_label`
- `status`
- `appealed`
- `appeal_reasoning`

The raw submitted text is intentionally **not** stored, to avoid retaining
user content. The log provides traceability for every attribution decision and
is updated in place when an appeal is filed.

# Anticipated Edge Cases

1. **Repetitive short poem** — intentional word repetition and simple vocabulary
   can make stylometry lean AI. The LLM weight and the 0.90 threshold guard
   against a false definitive verdict.
2. **Polished technical article** — highly structured human writing can push the
   Groq classifier toward AI. The combined score often lands in the Uncertain
   band rather than a definitive AI label.
3. **AI text heavily edited by a human** — the semantic signal may detect AI
   patterns while stylometry resembles human writing, typically producing an
   Uncertain result.
4. **Very short input** — with too few words/sentences, stylometry returns a
   neutral 0.50 so it neither dominates nor misleads the decision.
5. **Groq unavailable** — the endpoint returns HTTP 500 with
   `{"error": "Groq service unavailable"}` instead of crashing.
